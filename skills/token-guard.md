# Token Guard 技能

## 概述
LLM 调用限流和监控系统，防止 API 滥用和成本超支。

## 背景
2026-03-28 尝试集成失败，OpenClaw 不支持 LLM 级别 hook，需要联系作者添加支持。

## 设计目标
1. **限流控制**: 限制每分钟/每小时调用次数
2. **成本监控**: 跟踪 Token 使用和费用
3. **异常检测**: 识别异常调用模式
4. **自动拦截**: 超过阈值时自动拒绝调用

## 架构设计

### 拦截层
```
LLM 请求 → Token Guard → 限流检查 → 允许/拒绝 → LLM API
```

### 监控指标
1. **调用频率**: 请求/分钟，请求/小时
2. **Token 使用**: 输入/输出/总计 Token
3. **成本估算**: 基于模型定价
4. **响应时间**: API 延迟监控

## 限流策略

### 1. 频率限制
- **默认**: 60 请求/分钟，1000 请求/小时
- **可配置**: 按模型/用户/应用调整
- **突发**: 允许短期超出，平滑限制

### 2. Token 限制  
- **会话级**: 单次调用最大 Token
- **日级**: 每日 Token 上限
- **月度**: 月度预算控制

### 3. 成本控制
- **预算预警**: 达到80%预算时警告
- **自动停止**: 达到100%预算时停止
- **按需调整**: 动态调整限制

## 实现方案

### 方案A: 代理拦截（失败）
- **尝试**: 本地代理监听 LLM 流量
- **问题**: OpenClaw 内部路由复杂
- **状态**: ❌ 失败

### 方案B: Gateway Hook（需要支持）
- **需求**: OpenClaw 添加 LLM 调用 hook
- **位置**: `src/acp/translator.ts` 的 `chat.send` 方法
- **状态**: ⏳ 等待作者支持

### 方案C: 外部监控
- **实现**: 日志分析和告警
- **限制**: 只能监控，不能拦截
- **状态**: ✅ 可行但有限

## 技术挑战

### 1. OpenClaw 架构限制
- LLM 调用在 `translator.ts` 中直接进行
- 无插件系统拦截点
- 需要源码修改

### 2. 并发管理
- 多 agent 并发调用
- 分布式限流需求
- 状态同步问题

### 3. 准确计量
- Token 计数准确性
- 成本计算模型差异
- 实时性要求

## 临时解决方案

### 1. 日志监控
```bash
# 监控 LLM 调用日志
tail -f ~/.openclaw/logs/gateway.log | grep "LLM request"
```

### 2. 手动限流
- 控制并发子 agent 数量
- 避免批量重任务
- 定期检查使用情况

### 3. 预算提醒
- 设置每日使用提醒
- 手动暂停重任务
- 优先使用低成本模型
